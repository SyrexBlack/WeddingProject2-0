## Детали фаз

### Phase 1: Foundation (Фундамент)

**Цель:** Инициализированный проект Next.js 15 с настроенными шрифтами, типами, константами и базовыми UI-компонентами
**Зависит от:** Ничего (первая фаза)
**Требования:** FOUND-01, FOUND-02, FOUND-03, FOUND-04, FOUND-05
**Планы:** 3 plans

Plans:
- [x] 01-01-PLAN.md — Project scaffold, fonts, global styles, TypeScript types
- [x] 01-02-PLAN.md — Wedding data constants, base UI components
- [x] 01-03-PLAN.md — Page assembly, SectionDots navigation, visual verification

**Критерии успеха (что должно быть ИСТИНОЙ):**
1. `npm run dev` запускает проект без ошибок — главная страница рендерится в браузере
2. Шрифты (заголовочный + текстовый) загружаются без мерцания (FOUT) на первом визите
3. TypeScript-типы (`WeddingData`, `TimelineEvent`, `RSVPFormData`) экспортируются и импортируются корректно в любом компоненте
4. `constants.ts` содержит все данные свадьбы (имена, дата, место, программа, дресс-код) с полной типизацией
5. Базовые UI-компоненты (`Container`, `Button`, `Card`, `AnimatedSection`) рендерятся на странице без ошибок

---

### Phase 2: Static Sections (Статические секции)

**Цель:** Все информационные секции страницы отображают контент из constants.ts
**Зависит от:** Phase 1 (типы, константы, UI-компоненты)
**Требования:** CONT-01, CONT-03, CONT-04, CONT-05, CONT-07
**Планы:** 2 plans

Plans:
- [ ] 02-01-PLAN.md — Hero, Info, Dresscode, Footer section components
- [ ] 02-02-PLAN.md — Timeline zigzag section + page.tsx final assembly

**Критерии успеха (что должно быть ИСТИНОЙ):**
1. Hero-секция показывает имена пары, дату свадьбы и полноэкранное фоновое фото
2. Информационные карточки отображают «Где» и «Когда» с иконками Lucide
3. Секция дресс-кода показывает описательный текст и цветовую палитру свадьбы
4. Таймлайн программы дня отображает все этапы торжества с временем и описанием (вертикальная линия)
5. Footer показывает контактную информацию и копирайт

---

### Phase 3: Dynamic Components (Динамические компоненты)

**Цель:** Работающие клиентские компоненты — countdown, галерея, анимации при скролле
**Зависит от:** Phase 1 (UI-компоненты, типы), Phase 2 (секции для оборачивания анимациями)
**Требования:** CONT-02, CONT-06, INTER-05
**Планы:** TBD

**Критерии успеха (что должно быть ИСТИНОЙ):**
1. Countdown корректно отсчитывает дни, часы, минуты и секунды до свадьбы (правильный timezone, обновление в реальном времени)
2. Галерея показывает сетку из 6-12 фото с lazy loading, при клике открывается лайтбокс с навигацией
3. Секции появляются с fade-in анимацией при скролле (Framer Motion `whileInView`)
4. Анимации плавные на мобильных устройствах — нет рывков, используются только `transform`/`opacity`

---

### Phase 4: External Integrations (Внешние интеграции)

**Цель:** Работающая карта Яндекс.Карт и RSVP-форма с отправкой данных через EmailJS
**Зависит от:** Phase 1 (типы, env-переменные, базовая структура)
**Требования:** INTER-01, INTER-02, INTER-03, INTER-04
**Планы:** TBD

**Критерии успеха (что должно быть ИСТИНОЙ):**
1. Карта Яндекс.Карт (iframe) показывает место проведения свадьбы с маркером
2. Кнопка «Построить маршрут» открывает навигатор (Яндекс.Навигатор/Google Maps) с координатами места
3. RSVP-форма принимает имя, статус участия (приду/не приду/+1), пожелания — и отправляет данные через EmailJS
4. Форма показывает визуальную обратную связь: индикатор загрузки, сообщение об успехе, сообщение об ошибке

---

### Phase 5: Polish & Deploy (Полировка и деплой)

**Цель:** Оптимизированный, задеплоенный сайт с рабочими OG-тегами в мессенджерах
**Зависит от:** Phase 2, Phase 3, Phase 4 (весь функционал должен быть готов)
**Требования:** DEPLOY-01, DEPLOY-02, DEPLOY-03, DEPLOY-04, DEPLOY-05
**Планы:** TBD

**Критерии успеха (что должно быть ИСТИНОЙ):**
1. При отправке ссылки в Telegram и WhatsApp отображается карточка с фото и заголовком (OG-теги)
2. Lighthouse Performance score > 90 на мобильном
3. Сайт корректно отображается на экранах от 320px ширины — нет горизонтального скролла, текст читаем
4. Сайт доступен по публичному URL на Vercel с HTTPS
5. Все изображения оптимизированы: WebP формат, lazy loading, blur placeholder через `next/image`

**Цель:** Дифференциаторы, делающие приглашение запоминающимся — после стабильного MVP
**Зависит от:** Phase 5 (стабильный задеплоенный сайт)
**Требования:** ENHANCE-01, ENHANCE-02, ENHANCE-03, ENHANCE-04, ENHANCE-05, ENHANCE-06, ENHANCE-07
**Планы:** TBD

**Критерии успеха (что должно быть ИСТИНОЙ):**
1. Персонализация через URL-параметры работает — гость видит своё имя в приглашении (`?to=base64(Имя)`)

## Граф зависимостей

```
Phase 1 (Фундамент)
  ├─→ Phase 2 (Статические секции)
  │     └─→ Phase 3 (Динамические компоненты) ─┐
  └─→ Phase 4 (Внешние интеграции) ─────────────┤
                                                 ▼
                                  Phase 5 (Полировка и деплой)
                                                 │
                                                 ▼
                                  Phase 6 (Расширенные фичи)
```

- **Phase 1 → Phase 2:** Секции используют типы, константы и UI-компоненты из фундамента
- **Phase 2 → Phase 3:** Анимации оборачивают существующие секции, галерея встраивается в страницу
- **Phase 1 → Phase 4:** Интеграции используют env-переменные и базовую структуру проекта
- **Phase 2, 3, 4 → Phase 5:** Полировка и деплой требуют готового функционала
- **Phase 5 → Phase 6:** Расширения добавляются к работающему задеплоенному сайту
